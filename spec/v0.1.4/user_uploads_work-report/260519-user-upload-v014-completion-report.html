제품 목표

유저 업로드는 공식 공용 위키 corpus를 대체하는 기능이 아니다. 사용자가 폐쇄망 운영 자료, 교육 PDF, 사내 절차서를 올렸을 때 원문 구조를 최대한 보존한 private 문서로 저장하고, 뷰어에서 확인하며, 같은 사용자의 챗봇 질의에 근거로 사용할 수 있게 하는 기능이다.

v0.1.4의 완료 기준은 “문서가 예쁘게 실패했다고 보이는 것”이 아니라, 실제 업로드 자료가 저장, 파싱, 이미지 에셋화, DB 저장, Qdrant 인덱싱, 뷰어 확인, active document 기반 챗봇 질의까지 이어지는 것이다.

현 시점 완료 조건

Upload

파일 저장과 진행 로그

업로드 요청은 8단계 stage event로 기록된다. 이미지 에셋 저장과 OCR JSON 저장은 개별 progress event로 표시된다.

완료
Parser

PDF 원문 구조 보존

PyMuPDF 텍스트, drawing rect, 이미지 bbox를 활용해 제목, 문단, 표, 코드블록, 이미지 배치를 재구성한다.

완료
Viewer

HTML 문서 뷰어

업로드 문서는 Markdown dump가 아니라 HTML reader로 열린다. 코드블록은 복사, 줄바꿈, 긴 코드 접기 제어를 제공한다.

완료
RAG

챗봇 검색 연계

같은 owner의 active document 질문은 업로드 문서 chunk를 검색하고 citation을 viewer path로 연결한다.

완료

현재 8단계 파이프라인

1

요청 접수

프런트가 파일, owner, visibility, repository context를 업로드 API에 보낸다.

주요 코드: PlaybookLibraryPage.tsx, upload_api.py

2

원본 저장

원본 파일은 object storage 아래 source 파일로 저장된다. 재업로드는 기본 force reingest로 최신 결과를 만든다.

경로: storage/uploads/sources/<sha>/파일명

3

문서 파싱

PDF 텍스트, 표, 이미지, 회색 코드 박스 rect를 추출해 layout block으로 직렬화한다.

주요 코드: document_parsing.py

4

청크 생성

문서 block을 검색 가능한 chunk로 나누고 section path, heading, source anchor를 유지한다.

주요 함수: build_document_chunks

5

DB 저장

문서 source, version, parse job, parsed document, block, asset, chunk가 PostgreSQL에 저장된다.

테이블: document_sources, parsed_documents, document_chunks, document_assets

6

에셋 저장

PDF 내부 이미지는 파일로 materialize되고, OCR 결과는 검색 보조 JSON으로 저장된다. 뷰어 본문에는 OCR dump를 직접 노출하지 않는다.

경로: storage/uploads/assets/<document_source_id>/images, ocr

7

Qdrant 인덱싱

chunk embedding을 Qdrant에 올리고 qdrant_index_entries에 point mapping을 저장한다.

주요 코드: qdrant_indexer.py

8

스코프 확인과 완료 판정

owner, repository, document scope를 확인하고 ready stage를 기록한다. 답변 품질 검수는 별도 검증으로 남긴다.

주요 코드: retrieval/access_scope.py, retriever_plan.py

현재 남은 리스크

이번 작업은 “원문에 시각적 코드 박스가 있는 PDF”를 훨씬 안정적으로 처리한다. 다만 모든 PDF가 같은 방식으로 만들어지는 것은 아니므로, 회색 배경이 없거나 폰트/좌표 정보가 불량한 문서는 여전히 휴리스틱 판단에 의존한다.

또한 OCR은 이미지 설명과 검색 보조 맥락으로는 유용하지만, 사람용 뷰어 본문에 OCR dump를 그대로 붙이는 방식은 금지해야 한다. 원문 이미지, 텍스트, 코드, 표의 사람용 표현과 검색용 보조 텍스트는 분리되어야 한다.

예정 중인 고도화

P1

레이아웃 classifier 고도화

  • 회색 배경 외에도 border, font family, x/y alignment, line spacing을 종합해 code/prose/table을 판단한다.
  • page split code block은 시각 박스와 language continuity를 함께 보고 더 정교하게 이어붙인다.
  • ambiguous block은 rule만으로 결론내지 않고 Gemma4 기반 judge 후보로 넘긴다.
P1

NotebookLM식 source grounding 강화

  • active document 질문은 해당 문서 chunk를 우선 검색하고, 필요할 때만 공식 문서로 확장한다.
  • 문서명, 섹션명, 원문 키워드를 query expansion에 반영해 “문서 안 질문”을 더 안정적으로 처리한다.
  • 답변 전 citation coverage가 낮으면 “근거 없음”이 아니라 document-local retry를 수행한다.
P2

에셋 관리와 정리 정책

  • 업로드 삭제 시 source, report, assets, qdrant point가 모두 정리되는 현재 정책을 UI에서 더 명확히 노출한다.
  • 오래된 실패 업로드, stale report, orphan asset을 주기적으로 청소하는 maintenance task를 둔다.
  • 저장 용량과 문서별 asset 수를 관리 화면에서 확인하게 한다.
P2

진행 로그 UX

  • 문서 파악 후 이미지 개수, OCR 개수, 에셋 저장 개수를 progress bar로 표시한다.
  • 단계별 duration이 비정상적으로 빠르거나 느릴 때 설명 로그를 남긴다.
  • 완료 판정에는 “기본 인덱싱 완료”와 “답변 품질 검수 완료”를 분리해서 보여준다.
P3

위키문서화 레이어

  • corpus/data의 공용 위키 구조를 그대로 쓰지 않고, manifest, asset, relation index, viewer route라는 모양만 private scope로 재현한다.
  • 업로드 문서에서 자동 목차, key concept, command index, image relation을 생성한다.
  • J의 v0.1.5 작업에서는 private upload를 공식 corpus 승격 후보로 전환하는 명시적 승인 flow를 설계한다.
P3

평가 자동화

  • 업로드 직후 문서별 질문지를 자동 생성하고, citation hit 여부와 답변 정확도를 smoke test로 남긴다.
  • 대표 자료 02, 03, 07, 11, 12, 13번 PDF를 regression corpus로 고정한다.
  • 원문 대비 코드블록, 표, 이미지 배치 손실률을 측정하는 visual QA를 추가한다.
