현재 판단

파서와 뷰어는 레이아웃 기반 분류로 개선되었고, 실제 업로드 문서의 DB/HTML 산출물도 대체로 정상화되었다. 다만 챗봇 답변 생성 단계에서 긴 명령어 URL을 다시 시각적 줄바꿈하는 문제가 남아 있어, “업로드 문서 품질이 곧 답변 품질까지 완전 보장된다”고 말하면 안 된다.

열린 문제

OPEN Answer Formatting RAG citation은 잡힘

T-001. 챗봇 답변에서 긴 PowerShell URL이 다시 줄바꿈됨

DB와 뷰어 HTML에는 긴 URL이 한 줄로 저장되어 있는데, 챗봇 답변 생성 결과에서 openshift-pi / pelines.apps...처럼 시각적 줄바꿈이 다시 들어간 사례가 확인되었다.

Evidence
  • DB의 document_blocks.markdown에는 전체 URL이 한 줄로 존재한다.
  • 뷰어 API HTML에도 전체 URL 문자열이 포함된다.
  • /api/chat smoke에서 같은 코드블록이 답변으로 재생성될 때 줄이 나뉜 사례가 있었다.
Current Diagnosis
  • 원인은 PDF 파서나 DB 저장이 아니라 answer synthesis 또는 answer formatting 단계로 보는 것이 맞다.
  • 복사 가능한 명령어를 답변하는 경우에는 코드블록 내부 줄바꿈 보존 규칙을 더 강하게 걸어야 한다.
정상 원본:
smee --url https://smee.io/esRZDPmzzYd87BN8 --target https://pipelines-as-code-controller-openshift-pipelines.apps.ocp.test.com/

답변에서 관측된 문제:
smee --url https://smee.io/esRZDPmzzYd87BN8 --target https://pipelines-as-code-controller-openshift-pi
pelines.apps.ocp.test.com/
Acceptance Criteria
  • PowerShell 코드블록 답변에서 URL/명령어가 DB chunk 원문과 동일한 줄 구조를 유지한다.
  • 복사했을 때 명령어가 실행 가능한 문자열로 유지된다.
Suggested Next File Area
  • src/play_book_studio/answering/*
  • src/play_book_studio/http/server_chat.py
  • 답변 후처리 또는 코드블록 보존 프롬프트/formatter 테스트
OPEN Diagnostic Trap

T-002. 뷰어 API 진단 시 documentId가 아니라 viewer_path가 필요함

프론트는 업로드 문서 뷰어를 /uploads/documents/{document_source_id}/index.html 형태로 열고, /api/viewer-document도 viewer_path 파라미터를 요구한다. documentId로 호출하면 실제 제품 실패가 아니어도 viewer_path가 필요합니다. 에러가 난다.

Evidence
  • apps/web/src/lib/runtimeApi.ts의 loadViewerDocument(viewerPath)
  • apps/web/src/pages/PlaybookLibraryPage.tsx의 업로드 문서 viewer path 생성
  • src/play_book_studio/http/server_routes_viewer.py의 업로드 문서 viewer path regex
Risk
  • 테스트/수동 검증자가 잘못된 API 계약으로 확인하면 “뷰어가 안 뜬다”고 오판할 수 있다.
  • 다음 작업자에게 진단 URL을 문서로 고정해야 한다.
정상 진단 URL:
/api/viewer-document?viewer_path=%2Fuploads%2Fdocuments%2F{document_source_id}%2Findex.html
OPEN Source of Truth

T-003. document_sources.metadata.viewer_path는 비어 있고 프론트가 경로를 계산함

최신 업로드 row를 확인했을 때 document_sources.metadata->'viewer_path'는 비어 있었다. 현재 화면은 이 값을 읽는 대신 document_source_id로 viewer path를 계산한다.

Impact
  • DB만 보는 사람은 “viewer_path 저장이 안 됐다”고 오판할 수 있다.
  • 검색 payload, citation, viewer route의 truth source가 분산되어 보인다.
Suggested Decision
  • v0.1.4에서는 계산식 유지가 가능하지만, handoff 문서에 공식 viewer path 규칙을 명시한다.
  • v0.1.5에서 DB metadata에도 canonical viewer path를 저장할지 J와 협의한다.

F-001. 코드블록 내부 ## 예 ...가 제목처럼 렌더링되던 문제

레이아웃 기반 code block으로 먼저 분류한 뒤 Markdown을 만들도록 바꿔, 코드블록 내부의 ## 라인이 heading으로 승격되지 않게 했다.

FIXED IN WORKTREE Parser

F-002. URL이나 일반 문장이 코드로 과분류되던 문제

https:// 같은 URL scheme은 YAML의 key: 패턴으로 보지 않도록 제외했고, 코드성 라인 수와 레이아웃 정보를 함께 보도록 조정했다.

FIXED IN WORKTREE Viewer

F-003. OCR 결과가 뷰어 본문에 직접 노출되던 문제

현재 검증 대상 문서의 viewer HTML에서는 OCR 결과 문자열이 노출되지 않는다. OCR 텍스트는 검색 보조 맥락으로만 쓰고, 사람용 본문에는 원문 구조를 우선한다는 기준을 유지해야 한다.

FIXED IN WORKTREE Parser

F-004. 페이지가 갈라진 회색 코드 박스가 여러 코드블록으로 찢어지던 문제

PyMuPDF drawing rect에서 원문 회색 코드 박스를 감지해 visual_group으로 묶는다. 같은 시각 박스 안의 namePrefix, 주석, 다음 페이지 images 조각은 하나의 YAML 코드블록으로 유지하고, 회색 박스 밖의 Project Name, INCLUDE / EXCLUDE, path 설명문은 문단으로 둔다.

FIXED IN WORKTREE Retrieval

F-005. active upload 질문이 외부 제품명 때문에 검색 전 차단되던 문제

ArgoCD 같은 단어는 공식 OCP 코퍼스 기준으로는 외부 제품 차단 대상이지만, 사용자가 업로드 문서를 명시적으로 선택한 상태에서는 그 문서 내부를 먼저 검색해야 한다. active document/repository scope가 있으면 unsupported product gate를 우회하도록 조정했다.
