User Upload Runtime Guide

유저 업로드 8단계 실행 안내도

이 문서는 /playbook-library/repository의 유저 업로드가 현재 코드에서 어떤 순서로 실행되는지, 각 단계가 실제로 어떤 함수와 외부 서비스를 쓰는지, 그리고 제품 요구사항과 현재 구현 사이의 gap이 무엇인지 정리한다.

8 backend events 서버가 NDJSON stream으로 내보내는 실제 stage event 수
5 visible cards 현재 프론트 진행 카드가 공식 단계로 그리는 수
Text RAG 현재 구현이 주로 보장하는 완료 범위
Wiki asset gap PDF 페이지 이미지, OCR, diagram asset 보존은 별도 계약 필요
Company LLM OCR LLM_ENDPOINT 기반 Gemma 4 OCR/VL을 파싱 단계의 기본 경로로 둔다

화면 5단계와 8단계 매핑

화면 카드 수는 제품 UX용 압축이고, 백엔드 stage event는 진단용 실제 실행 단위다. parse는 내용 검증 단계이므로 원본 저장 카드에 숨기지 않는 편이 안전하다.

화면 카드 백엔드 stage 완료 의미
1. 원본 저장 received + store 요청을 받았고 원본 파일 bytes가 서버 storage에 저장됨. 아직 본문 추출 성공은 아님.
2. 문서 파싱 parse 본문 markdown, block, asset 후보가 만들어짐. OCR/VL도 이 단계의 sub-step으로 들어와야 함.
3. 청크 생성 chunk RAG 검색 단위인 chunk가 생성됨. 아직 viewer와 검색 인덱스에는 고정되지 않음.
4. DB 저장 persist PostgreSQL에 source, parsed document, block, chunk, asset row가 저장됨. 이 시점부터 업로드 viewer가 본문을 읽을 수 있음.
5. Qdrant 인덱싱 index + scope + ready embedding/Qdrant 반영과 owner/repository scope 확인까지 끝남. 텍스트 검색 가능 여부는 indexed_count로 판단.

뷰어 기준 완료점

parse가 끝나면 본문은 메모리에만 있고, persist가 끝나야 viewer route가 PostgreSQL에서 읽어 화면에 보여줄 수 있다. 단, 현재 viewer는 원본 PDF 페이지 복원이 아니라 parsed_documents.markdown 기반의 parsed view다.

저장 위치

업로드 결과는 하나의 파일에 전부 저장되지 않는다. 원본 파일은 로컬 storage에 남고, 파싱 결과와 검색 단위는 PostgreSQL/Qdrant에 나뉘어 저장된다.

데이터 로컬/컨테이너 위치 설명
원본 업로드 파일 C:\Users\soulu\cywell\OCPOps-PlayBookStudio\storage\uploads\sources\{upload_id}\{filename}
/app/storage/uploads/sources/{upload_id}/{filename}		 _store_uploaded_file()이 저장한다. Docker compose에서 ./storage:/app/storage로 bind mount된다.
업로드 리포트 C:\Users\soulu\cywell\OCPOps-PlayBookStudio\storage\uploads\reports\{document_source_id}\ingestion-report.json stage event, block/chunk/index count, quality gate 상태를 저장한다.
파싱 본문과 block PostgreSQL: parsed_documents, document_blocks viewer가 읽는 본문 source. 별도 markdown 파일로 저장되지 않는다.
검색 chunk PostgreSQL: document_chunks RAG 검색의 원문 단위. viewer diagnostic에도 사용된다.
이미지/asset metadata PostgreSQL: document_assets + storage asset key 현재 PDF 경로는 asset이 0개일 수 있다. PDF page image/OCR 구현 후 이 영역이 중요해진다.
벡터 인덱스 Qdrant collection: openshift_docs document_source_id, repository_id, owner_user_id payload filter로 검색 scope를 잡는다.
업로드 viewer /uploads/documents/{document_source_id}/index.html 디스크 HTML 파일이 아니라 PostgreSQL에서 동적으로 렌더링되는 route다.

corpus/data에서 가져올 계약

기존 위키화 산출물은 user upload를 그대로 저장할 위치가 아니라, v0.1.5 구현자가 따라야 할 viewer/asset/relation 구조의 샘플이다. private user 문서를 corpus/data 공용 폴더에 섞으면 access boundary가 흐려지므로, 모양만 가져오고 저장 truth는 PostgreSQL과 storage/uploads에 둔다.

corpus/data 구조 현재 역할 user upload에 적용할 방식
wiki_runtime_books/active_manifest.json 책 slug, title, source ref, viewer path, runtime markdown path를 묶는 manifest. document_sources와 parsed_documents.metadata에 upload document manifest equivalent를 둔다. 파일 JSON을 공용 manifest에 바로 추가하지 않는다.
wiki_assets/{group}/{book_slug}/*.png 공식 위키 figure 이미지의 실제 파일 저장소. PDF page image와 embedded image는 storage/uploads/assets/{document_source_id}/...에 저장하고 document_assets.storage_key로 연결한다.
wiki_relations/figure_assets.json figure caption, asset URL, figure viewer path, source file, section hint, related entities를 연결. document_assets와 asset metadata에 caption/OCR/VL 결과, page number, viewer path, section hint를 저장한다.
wiki_relations/diagram_assets.json diagram과 일반 figure를 구분하고 semantic diagram으로 노출. Gemma 4 OCR/VL 결과로 asset_type, diagram_type, description_status를 남기는 계약을 둔다.
wiki_relations/entity_hubs.json entity 중심으로 관련 책, 다음 읽을 경로, figure를 묶음. v0.1.4에서는 구현하지 않고, v0.1.5에서 corpus enrichment 또는 relation table 후보로 넘긴다.
/playbooks/wiki-assets/*, /wiki/figures/* static asset serve와 figure detail viewer route. private upload용 route는 /uploads/documents/{document_source_id}/assets/{asset_id} 계열로 분리해야 한다.
v0.1.4 판단

이번 버전에서 할 일은 full wiki runtime을 새로 만드는 것이 아니라, upload pipeline을 UI, viewer, chatbot까지 닫힌 루프로 완성하고, user upload가 나중에 같은 위키 경험으로 승격될 수 있게 page image, OCR/VL, asset relation의 저장 계약을 분명히 남기는 것이다.

추천 방식: DB-backed wiki projection

더 좋은 방법은 corpus/data의 폴더를 복제하는 것이 아니라, 현재 DB와 storage/uploads에 저장된 private upload를 viewer가 필요할 때 위키형 구조로 투영하는 것이다.

1. 저장 truth는 DB와 scoped storage

원본, 파싱 본문, block, chunk, asset은 기존 업로드 테이블과 storage/uploads에 저장한다.

핵심 이유

private upload는 owner/repository/visibility가 붙어야 하므로 공용 corpus/data manifest에 직접 섞지 않는다.

필수 row

document_sources, parsed_documents, document_blocks, document_assets, document_chunks.

필수 파일

원본 PDF와 page/image asset bytes는 storage/uploads 아래에 둔다.

2. viewer는 manifest를 즉석 projection

정적 JSON 파일을 만들지 않아도 viewer route가 DB에서 manifest equivalent를 구성할 수 있다.

Book equivalent

document_source_id를 slug처럼 쓰고, title/filename/repository/owner를 viewer header로 만든다.

Asset equivalent

document_assets rows를 figure_assets.json의 item처럼 렌더링한다.

Relation equivalent

v0.1.4에서는 page, section, asset 연결까지만 보장하고 entity hub는 v0.1.5 enrichment로 넘긴다.

3. chat은 같은 scope로 검증

위키형 viewer와 RAG 검색이 같은 document_source_id를 바라보는지 확인한다.

UI

Repository list에서 업로드 문서가 보이고 viewer/action이 연결된다.

Viewer

/uploads/documents/{document_source_id}/index.html이 parsed body와 asset diagnostics를 보여준다.

Chat

답변 citation이 같은 document_source_id 또는 viewer path를 포함한다.

v0.1.4 완료 경계

v0.1.5에서 J가 이어받기 전에, 현재 버전은 user upload runtime pipeline을 제품에서 확인 가능한 상태로 끝낸다. 다만 full wiki/corpus promotion까지 밀어붙이지는 않는다.

업로드한 자료가 Repository UI에 표시되고, viewer에서 파싱 본문을 확인할 수 있고, Ask this document 또는 동일 scope의 Studio Chat에서 업로드 문서를 citation으로 쓰는 답변이 나오는지 확인한다. 이때 stage count, 저장 위치, OCR/VL provider, 현재 PDF asset gap도 함께 드러나야 한다.

v0.1.5로 넘길 것

현재 팩트

8단계는 화면 카드 수가 아니라 백엔드가 실제로 emit하는 stage event 기준이다. 프론트는 그중 일부만 시각화한다.

backend: received/store/parse/chunk/persist/index/scope/ready frontend card: store/parse/chunk/persist/index not guaranteed: full PDF image wiki viewer
성공의 최소 의미

chunk_count > 0이고 indexed_count > 0이면 텍스트 기반 RAG 검색은 가능하다.

성공이 아닌 것

basic_index_ready는 답변 품질 검증 완료나 원본 위키 복원을 의미하지 않는다.

제품 목표

PDF 원본 페이지, 이미지, OCR/VL 설명, 위키 viewer, RAG citation까지 같은 문서 계약으로 묶어야 한다.

OCR/VL Provider 결정

PDF 이미지와 스캔 페이지를 읽는 경로는 로컬 OCR을 기본값으로 두지 않는다. 회사 LLM 서버가 OpenAI-compatible vision 요청을 받아 처리하는 것을 파싱 단계의 기본 계약으로 둔다.

환경 변수

LLM_ENDPOINT=http://cllm.cywell.co.kr/v1, LLM_MODEL=gemma-4-26b-a4b-it-awq-8bit를 runtime 설정의 source of truth로 쓴다.

현재 코드 hook

upload_api.py는 build_qwen_image_describer(settings)를 넘기고, 이 client는 /chat/completions에 base64 image payload를 보낸다.

현재 gap

PDF 파서가 페이지 이미지나 내부 이미지를 DocumentAsset으로 만들지 않으면 LLM_ENDPOINT가 있어도 OCR/VL 호출은 발생하지 않는다.

이름 부채

코드의 QwenVisionClient, qwen_model, qwen_status는 현재 모델을 뜻하는 이름이 아니라 과거 구현명이 남은 것이다. 실제 provider 판정은 환경 변수의 LLM_ENDPOINT와 LLM_MODEL로 해야 한다.

8단계 실행 지도

아래 단계명은 백엔드 stage event 이름 그대로다. 각 단계는 사용자가 보는 UI 문구와 다를 수 있다.

received — 요청 접수

업로드 API가 요청을 받고 NDJSON stream을 시작한다.

실행 진입점

handle_upload_ingest_stream()가 stream을 열고 build_upload_ingest_response()를 호출한다.

쓰는 도구

Python HTTP handler, multipart payload, NDJSON event writer.

처리 결과

아직 파일 분석은 하지 않는다. 사용자는 “서버가 업로드 요청을 받았다”는 신호만 받는다.

event: running event: done

store — 원본 파일 저장

업로드된 파일 bytes를 서버 object storage 아래에 저장한다.

실행 함수

_store_uploaded_file()이 파일명을 정리하고 storage path를 만든다.

쓰는 도구

Python filesystem write, Path.write_bytes(), load_settings().

처리 결과

uploads/sources/{upload_id}/{filename}에 원본 파일이 저장되고 storage_key가 생긴다.

output: source_path output: storage_key output: byte_size

parse — 문서 파싱

원본 파일을 markdown, block, asset 후보로 변환한다.

실행 함수

parse_upload_document(). PDF는 _convert_pdf_to_markdown()을 탄다.

쓰는 도구

PDF text: PyMuPDF fitz 우선, 품질이 낮거나 실패하면 pdfplumber fallback 경로. PDF image/OCR: page image asset을 만든 뒤 LLM_ENDPOINT의 Gemma 4 vision 요청으로 처리해야 한다.

처리 결과

ParsedUploadDocument, DocumentBlock, markdown, warnings, metadata가 만들어진다.

info: format detected info: extractor progress guard: block_count > 0 fail: empty parse

chunk — 검색 청크 생성

파싱된 block을 RAG 검색 단위로 나눈다.

실행 함수

build_document_chunks()가 heading, size, overlap 기준으로 block을 묶는다.

쓰는 도구

내부 Python splitter. 별도 외부 API는 호출하지 않는다.

처리 결과

DocumentChunk마다 embedding_text, section, page range, asset ids가 붙는다.

guard: chunk_count > 0 fail: no searchable chunk

persist — PostgreSQL 저장

문서, 파싱 결과, block, asset, chunk를 DB에 저장한다.

실행 함수

persist_parsed_upload_document()와 build_parsed_document_rows().

쓰는 도구

psycopg, PostgreSQL transaction, repository/document upsert.

처리 결과

document_source_id, repository_id, parsed_document_id, chunk rows가 확정된다.

table: document_sources table: parsed_documents table: document_chunks visibility: private_user or workspace_shared

index — Qdrant 인덱싱

DB chunk를 embedding vector로 만들고 Qdrant collection에 upsert한다.

실행 함수

index_pending_document_chunks()가 후보 조회부터 Qdrant 기록까지 처리한다.

쓰는 도구

EmbeddingClient가 remote embedding /embeddings를 호출하고, Qdrant REST API를 호출한다.

처리 결과

Qdrant point와 qdrant_index_entries가 생긴다. candidate_count, indexed_count가 응답에 들어간다.

output: indexed warning: partial failed: embedding or qdrant error

scope — 검색 범위 확인

이 문서가 어떤 owner/repository/document scope에 묶였는지 확인한다.

실행 로직

직전 단계 결과의 repository_id와 document_source_id 존재 여부를 확인한다.

쓰는 도구

외부 도구 없음. DB/Qdrant 추가 호출도 아니라 결과값 검증 이벤트에 가깝다.

처리 결과

나중에 Studio Chat이 activeRepositoryId, activeDocumentId로 필터링할 기준이 된다.

done: ids present warning: scope incomplete

ready — 기본 인덱싱 판정

검색 가능한 기본 텍스트 인덱싱인지, 답변 품질 검증까지 된 것인지 분리한다.

실행 함수

_basic_index_ready(), _answer_ready(), _apply_upload_readiness().

쓰는 도구

내부 판정 로직. 외부 API 호출 없음.

처리 결과

basic_index_ready, answer_ready, ready_for_chat가 응답과 report에 기록된다.

basic_index_ready: text search ready answer_ready: quality gate required ready_for_chat = answer_ready

8단계 이후의 채팅 연결

업로드 pipeline 자체와 Studio Chat RAG 검색은 별도 흐름이다. 버튼을 눌러 scope를 넘겨야 문서 단위 질문이 된다.

프론트가 넘기는 값

이 문서로 질문 버튼은 workspace.activeSourceId에 repository id를, workspace.activeDocumentId에 document source id를 저장한다.

검색이 쓰는 필터

Chat 요청은 active_repository_id, active_document_id를 보낸다. retrieval은 Qdrant filter에 document_source_id, repository_id를 넣는다.

현재 빠진 제품 계약

아래 항목은 8단계 이름만 봐서는 들어있어 보이지만, 현재 구현이 충분히 보장하지 않는 부분이다.

항목 현재 상태 제품적으로 필요한 상태
PDF 페이지 이미지 텍스트 추출 중심. 페이지를 이미지 asset으로 렌더링해 보존하는 계약은 없음. 각 페이지를 viewer asset으로 저장하고 페이지 단위로 탐색 가능해야 함.
이미지/OCR/VL LLM_ENDPOINT 기반 image describer hook은 있으나, 텍스트 PDF 내부 이미지까지 항상 asset화하지는 않음. PDF 내부 이미지/캡처/도식을 page/image asset으로 만든 뒤 Gemma 4 OCR/VL 설명을 생성하고 RAG embedding text에 반영.
위키 viewer 업로드 viewer는 parsed markdown과 chunk diagnostic을 보여주는 화면. corpus/data/wiki_assets, wiki_relations 같은 asset 관계를 가진 위키형 문서.
Chat-ready 문구 basic_index_ready와 ready_for_chat의 의미가 UI에서 섞일 수 있음. “텍스트 검색 가능”, “답변 품질 검수 전”, “검증된 답변 가능”을 분리해야 함.

검증 체크리스트

수정 후에는 아래 기준을 통과해야 “업로드가 RAG에 반영됐다”고 말할 수 있다.

  • 텍스트 PDF 업로드 후 block_count > 0, chunk_count > 0, index.indexed_count > 0가 응답에 나온다.
  • PostgreSQL의 document_sources, document_chunks, qdrant_index_entries가 같은 document_source_id로 이어진다.
  • Studio Chat이 activeDocumentId scope로 질문했을 때 citation에 업로드 문서의 viewer path가 포함된다.
  • PDF 내부 이미지/도식이 있는 경우 document_assets 또는 위키 asset 저장소에 남고, viewer에서 확인 가능하다.
  • 이미지/스캔 페이지가 있는 경우 parse metadata에 LLM_MODEL=gemma-4-26b-a4b-it-awq-8bit 기반 OCR/VL 결과가 남는다.
  • 스캔 PDF처럼 텍스트 추출이 불가능한 경우 “성공”이 아니라 OCR 필요 또는 파싱 실패로 명확히 떨어진다.

코드 위치 빠른 참조
