User Upload / Ingestion / Report

유저 업로드 문서 처리 단계와 실시간 리포트 설계

목적은 “문서를 올렸더니 챗봇에 반영됐다”를 말로만 주장하지 않고, 파일 저장부터 PostgreSQL 저장, Qdrant 인덱싱, 챗봇 질문 가능 상태까지 사용자가 직접 확인할 수 있게 만드는 것이다.

기준 시점: feat/docs_uploads_S_v1 브랜치의 유저 업로드 흐름 점검. 이 문서는 팀 공유용 결론 문서라기보다, 현재 문제를 이해하고 다음 구현 기준을 잡기 위한 작업 메모다.

현재 판단

업로드 기능의 아이디어는 좋지만, 기존 UI는 처리 과정을 증명하지 못해 가짜 진행처럼 보였다.

핵심 문제

실제 처리 단계와 화면의 불 들어오는 단계, 로그, 문서 확인 화면이 서로 정확히 맞물리지 않았다.

필요한 방향

단계 이벤트 기반 실시간 로그와 업로드별 영속 리포트를 제품 기능으로 고정해야 한다.

0. 지금 실제로 문제가 된 장면

  • 업로드 직후 Source Intake → Structured Wiki → Playbook Materialize → User Library Join 단계가 거의 한 번에 켜져 사용자가 “실제 처리 중인지, 그냥 불만 켠 것인지” 신뢰하기 어려웠다.
  • 로그에는 14/14 indexed, 23/23 indexed처럼 성공이 찍혔지만, 화면에서 그 인덱싱 결과가 어떤 문서/청크/검색 범위로 연결됐는지 바로 검증할 수 없었다.
  • 문서 확인 화면은 원본 PDF 전체 뷰어가 아니라 “파싱된 청크 일부를 보여주는 화면”에 가까웠다. 그래서 사용자는 정상 업로드인지, 파싱 누락인지 구분하기 어려웠다.
  • PostgreSQL 저장, Qdrant 인덱싱, 챗봇 질문 가능 상태가 모두 다른 의미인데 화면에서는 하나의 SUCCESS처럼 보였다.
  • 유저 업로드 문서는 개인 문서이므로 owner_user_id, repository_id, document_source_id 기준으로 검색 범위가 격리되어야 한다. 이 격리 상태도 UI와 리포트에서 확인 가능해야 한다.

결론: “빠르게 끝난 것” 자체가 항상 거짓은 아니지만, 현재 UI는 빠른 처리와 가짜 진행을 구분할 증거를 충분히 보여주지 못했다. 따라서 단계별 서버 이벤트, 처리 시간, DB/Qdrant 반영 수, 문서 범위 키를 한 화면에서 보여주는 리포트가 필요하다.

1. 업로드 문서가 챗봇에 반영되기까지의 단계

현재 구조상 유저 업로드 문서가 챗봇 검색 근거로 쓰이기까지는 최소 8단계가 있다.

UI 진행 이미지는 이 8단계를 임의로 6단계나 4단계로 압축하지 않는다. 사용자가 신뢰해야 하는 것은 “보기 좋은 요약”이 아니라 실제로 어떤 작업이 진행 중인지이므로, 화면의 단계와 서버 처리 단계는 같은 이름과 같은 순서로 맞춘다.

  1. 업로드 요청 접수 브라우저가 파일과 세션 owner 정보를 서버로 보낸다. 현재는 X-User 기반 owner hash를 사용한다.
  2. 원본 파일 저장 파일을 object storage 하위 uploads/sources/<upload_id>/<filename> 경로에 저장한다.
  3. 문서 파싱 PDF/DOCX/TXT 등에서 텍스트, block, asset, metadata, warning을 추출한다.
  4. 청크 생성 챗봇 검색에 사용할 단위로 문서를 나누고 embedding_text 입력을 만든다.
  5. PostgreSQL 저장 document_sources, parse_jobs, parsed_documents, document_blocks, document_chunks 등에 저장한다.
  6. Qdrant 인덱싱 청크를 embedding으로 변환해 Qdrant point로 넣고, qdrant_index_entries에 인덱싱 기록을 남긴다.
  7. 챗봇 검색 범위 연결 owner_user_id, repository_id, document_source_id로 검색 범위를 제한한다.
  8. 상태 확인 및 문서 검수 사용자가 업로드 결과, 파싱 결과, Qdrant 반영 여부, 챗봇 질문 가능 여부를 확인해야 한다.

2. 각 단계에서 사용자에게 보여야 하는 정보

단계 화면에 보여야 하는 정보 실패 시 표시해야 하는 내용

3. 실시간 리포트 저장 가능 여부

가능하다. 그리고 이 기능에는 필요하다. 업로드는 파일마다 파싱 품질과 인덱싱 결과가 크게 달라지므로, “성공” 한 줄이 아니라 단계별 결과 리포트가 있어야 사용자가 결과를 신뢰할 수 있다.

현재 가능한 것

  • /api/uploads/ingest/stream으로 단계 이벤트를 흘릴 수 있다.
  • 응답에는 store_ms, parse_ms, chunk_ms, persist_ms, index_ms, total_ms가 들어갈 수 있다.
  • DB에는 문서/청크/Qdrant index entry를 조회할 수 있는 근거 테이블이 있다.

현재 부족한 것

  • 단계별 리포트가 영속 저장되지 않는다. 새로고침 후 상세 처리 이력을 복원하기 어렵다.
  • parse_jobs.status와 상태 API의 ready 판정 값이 어긋날 수 있다.
  • Qdrant index 실패, retry, duration이 DB에 충분히 남지 않는다.
  • 로그는 이벤트가 아니라 임시 화면 로그 중심이라 제품 신뢰 근거로 약하다.

4. 로그창에 반영되어야 하는가?

반영되어야 한다. 업로드 UI의 로그창은 단순 개발자 로그가 아니라 사용자가 “지금 내 문서가 어떤 처리 중인지” 확인하는 핵심 제품 영역이다.

현재 기대 필요한 UI
문서가 서버로 올라가는지 확인 업로드 요청, 파일 크기, 저장 완료 로그
파싱이 제대로 되었는지 확인 blocks, chunks, tokens, warnings 표시
챗봇 검색에 들어갔는지 확인 Qdrant candidate/indexed 수, collection 표시
질문 가능한 상태인지 확인 ready_for_chat, active document/repository scope 표시

5. 리포트 기능과 성능 리포트 자동화가 필요한 이유

  • 유저 업로드 파일은 공식 문서처럼 구조가 일정하지 않다.
  • PDF는 텍스트 PDF, 이미지 PDF, 표 중심 PDF, OCR 필요 PDF 등 품질 편차가 크다.
  • 업로드 성공과 챗봇 검색 성공은 같은 말이 아니다.
  • 파싱 결과가 600 tokens 수준이면, 챗봇 근거로 쓰기 부족할 수 있다는 경고가 필요하다.
  • 시연/운영/디버깅에서 “왜 답을 못했는지”를 추적하려면 upload별 처리 리포트가 필요하다.

결론: 리포트 기능은 부가 기능이 아니라 유저 업로드 RAG의 신뢰를 만드는 핵심 기능이다.

6. 현재 작업 방식에 대한 판단

현재까지의 방식은 비효율적이고 사용자 입장에서는 불쾌하게 느껴질 수 있다. 이유는 “처리 성공”이라는 말을 했지만, 사용자가 확인할 수 있는 증거와 단계별 상태가 충분히 제공되지 않았기 때문이다.

  • 진행 UI가 실제 단계 이벤트가 아니라 프론트 추정으로 켜졌던 문제가 있었다.
  • 문서 확인 화면이 원본 문서 확인인지, 파싱 결과 확인인지 명확하지 않았다.
  • 업로드 성공, DB 저장 성공, Qdrant 인덱싱 성공, 챗봇 질문 가능 상태가 분리되지 않았다.
  • 문서 파싱 품질이 낮아도 “성공”처럼 보여 사용자가 오해할 수 있었다.

7. 개선 플랜

순서 작업 완료 조건
1 업로드 8단계 계약 고정 received/store/parse/chunk/persist/index/scope/result 단계와 status 값을 UI와 서버에서 같은 의미로 사용
2 8단계 시각 진행바를 서버 이벤트 기반으로 표시 프론트가 서버 이벤트를 받을 때만 해당 단계가 진행 중 또는 완료로 바뀜
3 Upload Report 패널 추가 8단계 stage timeline, ingestion receipt, search index summary 표시
4 문서 확인 명칭 분리 파싱 결과 확인과 원본 파일 확인을 분리
5 파싱 품질 경고 추가 tokens/chunks가 비정상적으로 낮으면 OCR/파싱 품질 경고 표시
6 영속 리포트 저장 업로드별 JSON/DB 기반 리포트를 새로고침 후에도 확인 가능
7 상태 API 정규화 parsed/succeeded/completed/ready ready 판정 불일치 제거

8. Acceptance Criteria

  • PASS 업로드 중 현재 단계가 화면과 로그에 동시에 표시된다.
  • PASS 시각 진행 단계는 실제 처리 8단계와 같은 순서와 같은 의미를 가진다.
  • PASS 업로드 완료 후 파일명, block 수, chunk 수, token 수, indexed 수가 한눈에 보인다.
  • PASS Qdrant 인덱싱 실패 시 “저장은 완료됐지만 검색 불가”로 분리 표시된다.
  • PASS 파싱 결과가 지나치게 적으면 품질 경고가 뜬다.
  • PASS 리포트는 새로고침 후에도 다시 열 수 있다.
  • CHECK 원본 문서 뷰어가 필요한 경우 파싱 결과 뷰어와 별도 버튼으로 분리한다.
