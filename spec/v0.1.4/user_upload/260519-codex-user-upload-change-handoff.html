Codex User Upload 변경분 인수인계

이 문서는 완료 보고서가 아니다. Codex가 user upload 작업 중 건드린 변경분과 잘못된 판단, 이어받는 작업자가 반드시 다시 검증해야 할 지점을 남기는 인수인계 문서다.

현재 판단

변경분은 바로 신뢰하면 안 된다. 특히 업로드 진행 UI는 실제 처리 단계보다 부풀려진 설계가 들어갔다.

검증 상태

관련 pytest와 frontend build는 통과했지만, 브라우저에서 실제 파일 업로드 end-to-end 검증은 완료되지 않았다.

이어받을 핵심

같은 파일이라도 owner/session이 다르면 private 문서로 업로드 가능해야 한다. 같은 owner 안에서만 중복 안내가 맞다.

1. 제일 중요한 결론

Codex가 만든 8단계 업로드 진행 UI는 제품 기준으로 문제가 있다. 실제 무거운 처리 단계는 대략 원본 저장, 파싱, 청크 생성, PostgreSQL 저장, Qdrant 인덱싱의 5단계다. received, scope, ready는 처리 단계가 아니라 요청/검증/상태 판정 이벤트에 가깝다.

따라서 다음 작업자는 8단계 UI를 그대로 이어받지 말고, 사용자에게 보이는 메인 진행 단계는 5단계 중심으로 다시 정리하는 것이 좋다. 보조 이벤트는 작업 로그나 상세 리포트에 내려야 한다.

2. Codex가 건드린 주요 파일

파일 건드린 내용 주의점
src/play_book_studio/http/upload_api.py 업로드 stream stage event, ingestion report 저장, basic_index_ready/answer_ready 분리, report 접근 전 owner/visibility 검사, Qdrant 인덱싱 상태별 warning/fail 처리를 추가했다. 이후 같은 파일 다른 owner 업로드를 허용하려고 scoped_document_sha256 기반 lookup도 연결했다. 핵심 파일이다. 브라우저 실제 업로드 검증 전까지 신뢰 금지. 현재 8단계 event 구조가 UI에 그대로 노출되면 과장된 진행 상태로 보일 수 있다.
src/play_book_studio/db/document_repository.py private user upload의 DB unique 충돌을 피하려고 scoped_document_sha256()을 추가했다. 원본 sha는 metadata.original_sha256에 남기고, DB sha256은 owner scope hash로 바뀌게 했다. find_document_source_by_sha에는 owner와 indexed_count 관련 반환값이 들어갔다. DB migration 없이 UNIQUE(workspace_id, sha256) 제약을 우회하는 방식이다. 의도는 맞을 수 있으나, J/다음 작업자가 반드시 schema 계약과 충돌 여부를 검토해야 한다.
src/play_book_studio/http/document_status_api.py 문서 상태를 ready=true로 과장하지 않고, basic_index_ready와 answer_ready=false로 분리했다. 방향은 맞지만, UI에서 이 값을 어떻게 보여주는지 실제 화면 검증 필요.
src/play_book_studio/db/qdrant_indexer.py user upload chunk의 fallback viewer path를 /uploads/documents/{document_source_id}/index.html#{chunk_id} 형태로 바꿨다. viewer 쪽 실제 anchor 이동이 되는지 확인 필요.
apps/web/src/lib/runtimeApi.ts upload stream API 타입, report 타입, basic_index_ready/answer_ready 필드, upload stream 호출 함수를 추가/확장했다. 타입 확장은 build 통과. 실제 서버 이벤트 소비는 화면 검증 필요.
apps/web/src/pages/PlaybookLibraryPage.tsx Library upload UI에 8단계 진행 UI, stage event 로그, report modal, 과장 표현 제거 일부를 반영했다. 가장 문제가 큰 파일이다. 8단계 표시가 실제 처리 단계처럼 보이므로 재설계 필요.
apps/web/src/pages/WorkspacePage.tsx Studio 업로드도 일반 endpoint 대신 upload stream endpoint를 타도록 바꿨다. 상태 배너도 basic_index_ready 기준으로 바꿨다. 실제 Workspace 업로드 UX와 Library upload UX가 같은 계약을 쓰는지 확인 필요.
docker-compose.yml ./storage:/app/storage 마운트를 추가했다. report JSON과 업로드 저장소가 컨테이너 재생성 후에도 남게 하려는 목적이다. 방향은 맞지만 배포 환경에서 volume 정책과 충돌 없는지 확인 필요.
tests/test_upload_api.py, tests/test_qdrant_indexer.py upload readiness 분리, zero candidate, duplicate owner scope, report 접근 차단, upload viewer path 관련 테스트를 추가/수정했다. 테스트는 현재 코드 계약을 고정할 뿐, 실제 브라우저 end-to-end 성공을 보장하지 않는다.

4. 실제로 잘못한 판단

잘못한 점 왜 문제인가 다음 작업자가 해야 할 일
8단계를 실제 처리 단계처럼 설계 received, scope, ready는 실질 처리보다 상태 이벤트에 가깝다. 화면에서 즉시 켜지면 가짜 진행처럼 보인다. 메인 진행 UI를 5단계로 줄이고, 보조 이벤트는 로그/리포트에 분리한다.
테스트 통과를 제품 검증처럼 말함 pytest/build 통과와 실제 PDF 업로드 UX 성공은 다르다. 브라우저에서 실제 PDF 업로드, DB row, Qdrant index, report JSON, Studio 검색 scope까지 확인한다.
같은 파일 다른 owner 중복을 처음엔 차단 사용자는 같은 PDF를 다시 올릴 수 있어야 한다. 개인 문서 격리를 이유로 업로드를 막으면 제품성이 나쁘다. owner별 private duplicate 허용을 확정한다. 현재는 scoped DB hash 방식이 들어갔으나 실사용 검증 전이다.
Docker/포트 상태를 늦게 확인 Failed to fetch는 backend가 죽어 있어도 발생한다. 기능 실패와 서버 미기동을 분리했어야 했다. 5176, 8765, 8080 역할을 먼저 고정하고 확인한다.

6. 다음 작업자가 먼저 해야 할 확인

  1. git diff로 현재 변경분을 먼저 읽고, 바로 커밋하지 않는다.
  2. Library upload 진행 UI를 8단계에서 5단계 중심으로 재정리할지 결정한다.
  3. 같은 파일 업로드 테스트를 두 가지로 나눈다: 같은 owner에서는 duplicate 안내, 다른 owner에서는 새 private 문서 등록.
  4. 업로드 후 document_sources, parsed_documents, document_chunks, qdrant_index_entries, report JSON을 같이 확인한다.
  5. Studio 질문 요청에 active_repository_id, active_document_id, owner/session scope가 실제로 들어가는지 확인한다.
  6. 답변 품질은 아직 검증 완료로 말하지 않는다. 현재 표현은 최대 기본 텍스트 인덱싱 완료까지만 허용한다.

9. 이어받는 사람에게 남기는 한 줄

이 변경분은 “완성된 user upload 기능”이 아니다. Codex가 실제 요구를 UI/리포트 구조로 과대 번역한 흔적이 있으므로, 변경분을 선별해서 살리고 반드시 실제 업로드 end-to-end로 다시 검증해야 한다.
