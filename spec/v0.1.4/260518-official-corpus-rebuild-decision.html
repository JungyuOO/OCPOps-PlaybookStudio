챗봇 검색 매칭 점검 내역

2026-05-18 / branch: feat/corpus_embedding_test_S_v.2 / head: d8beca9

문제

  • 챗봇의 답변이 잘못된 데이터를 향하거나 답을 못 찾는다는 피드백에 대한 조치를 취하기 위해 점검을 진행했습니다.
  • 답변 LLM 자체 문제인지, 검색 단계에서 근거 문서를 잘못 가져오는 문제인지 분리할 필요가 있었습니다.

한 일: 공식문서 검색 테스트셋 정리

  • OCP 운영자가 물어볼 만한 공식문서 기준 단독 질문 30건을 정리했습니다.
  • 각 질문마다 최소한 찾아와야 할 공식문서군(expected_book_slugs)을 active official 29권 기준으로 맞췄습니다.
  • 기존에 현재 corpus에 없는 expected가 섞이면 점수가 왜곡될 수 있어 해당 기준을 보정했습니다.

한 일: 검색 결과 top-k 확인

  • RERANKER_ENABLED=false 조건에서 retrieval-only 평가를 실행했습니다.
  • answer LLM, reranker, UI 답변 문장은 이번 검증에서 제외했습니다.
  • 각 질문마다 top1~top5에 기대 공식문서가 들어오는지 확인했습니다.
  • 문서명만 맞는 것이 아니라 실제 청크 텍스트가 질문 근거로 쓸 수 있는지도 따로 확인했습니다.

한 일: 검색이 어긋나는 원인 분리

  • 정답 데이터가 없는 문제인지 확인했습니다.
  • 청크나 embedding_text가 오염된 문제인지 확인했습니다.
  • 검색어/의도 신호가 넓어서 덜 직접적인 문서가 먼저 뜨는 문제인지 확인했습니다.
  • 평가셋 expected 기준 자체가 잘못된 문제인지 확인했습니다.

한 일: 검색 매칭 보정

  • 특정 질문/특정 chunk_id를 하드코딩하지 않았습니다.
  • 일반 패턴 기준으로 query signal과 scoring 보정을 적용했습니다.
  • 예: 덜 직접적인 일반 문서보다 질문 의도와 더 가까운 공식문서 청크가 먼저 올라오도록 조정했습니다.
  • 평가셋 expected 오류는 1건만 보정했습니다.

확인 결과

  • official 단독 질문형 30건 기준 expected book_slug hit@5: 30/30
  • top-k의 embedding_text / normalized_text 정제 경고: 0건
  • 정답 데이터가 아예 누락된 문제는 현재 핵심 원인으로 보이지 않습니다.
  • 남은 품질 검증은 답변 LLM, reranker, UI 답변 문장 단계에서 별도로 봐야 합니다.

추가로 확인한 데이터 상태

  • 원문 chunks.jsonl: 27,907 rows 보존
  • 4계층 text_layers.jsonl: 27,907 rows 존재
  • 검색용 embedding_chunks.jsonl: 25,881 rows 존재
  • 다만 산출물 재현성 측면에서 text_layers와 embedding_chunks를 현재 코드 기준으로 다시 생성해 맞추는 것이 안전하다고 판단했습니다.
