이 파일은 상세 검증 로그입니다. 청크/임베딩 텍스트 수정 흐름을 먼저 이해하려면 260517-embedding-cleanup-timeline.html을 먼저 보는 편이 좋습니다.

이 문서의 목적

이번 확인은 챗봇 답변 품질 평가가 아니라, 변경된 official corpus / embedding 데이터가 질문에 맞는 근거 source와 청크 텍스트를 검색 단계에서 가져오는지 확인하기 위한 데이터 검증입니다.

기존 18건 테스트에는 follow-up/context/ambiguous 성격이 섞여 있어 “처음 사용자가 할 법한 단독 질문” 검수에는 맞지 않았습니다. 그래서 이번에는 retrieval_official_data_validation_cases.jsonl 30건으로 다시 실행했습니다. 모든 질문은 context 없이 단독으로 이해되도록 작성했고, 각 질문에는 검증 의도(validation_focus)를 같이 남겼습니다.

이번 범위에서 answer LLM, reranker, UI 답변 문장은 제외했습니다. 따라서 결과는 “최종 답변이 좋다/나쁘다”가 아니라 “검색기가 근거 후보를 잘 가져왔는가”로 읽어야 합니다.

확인된 file-integrity 인접 오염은 embedding projection과 Qdrant 기준으로 보정했습니다. tail-only 청크 1건은 검색 대상에서 제외했고, 유효 명령어가 포함된 청크 1건과 AIDE 로그 예시 청크 1건은 의미 신호만 남기도록 정리했습니다.

평가셋에서는 현재 official book set에 없는 applications expected를 제거하고, bootstrap/maxUnavailable처럼 근거가 여러 official book에 흩어진 케이스는 expected 기준을 보정했습니다. 보정 후 retrieval-only 재실행 결과는 30건 중 27건이 expected book을 top5 안에 포함했고, top15의 embedding_text/normalized_text 정제 노이즈는 0건입니다.

다만 file-integrity 직접 명령어, web_console, install-bootstrap, mcp-maxUnavailable 일부는 정답 청크가 존재해도 상위 순위에 충분히 오르지 않는 모습이 남아 있습니다. 이 부분은 추가 코퍼스 삭제보다는 J의 query rewrite 수정분을 받은 뒤 같은 질문셋으로 전후 비교하면서, 질문셋 expected 기준과 검색 순위 기준을 함께 맞춰보는 것이 좋아 보입니다.
