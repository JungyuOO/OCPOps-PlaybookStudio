# 260517 Audit

## 목적

v0.1.4 official corpus 작업을 커밋/푸시한 뒤, "끝났다고 말해도 되는 범위"와 "아직 별도 검증이 필요한 범위"를 다시 점검한 기록이다.

이 문서는 상세 correction note가 아니라 전체 점검 요약이다. 상세 근거와 명령 출력은 `260517-official-corpus-correction.md/html`에 남겼다.

## 결론

이번 브랜치에서 말할 수 있는 완료 범위:

- official corpus의 `text` / `normalized_text` / `embedding_text` 역할 분리를 바로잡았다.
- `embedding_text`와 `normalized_text`에서 base64-like 출력 예시가 남는 문제를 제거했다.
- Qdrant payload의 검색/표시 계층을 검수했고, official docs 전체 scan에서 주요 오염 지표 0건을 확인했다.
- BM25 runtime은 표시용 `text`가 아니라 `normalized_text`를 우선 토큰화하도록 수정했다.
- docker compose seed의 stale refresh 경로는 payload만 바꾸지 않고 vector까지 다시 upsert하도록 수정했다.

아직 "제품 답변 품질까지 완료"라고 말하면 안 되는 범위:

- 실제 챗봇 답변 품질은 별도 평가셋으로 top-k 근거와 최종 답변을 확인해야 한다.
- J가 진행 중인 metadata/rank policy 적용 결과는 이 브랜치 단독으로 보장하지 않는다.
- 새 chunk size를 준 rechunk 작업은 아니다. chunk size 정책 변경은 별도 협의가 필요하다.
- 반복 표 chunk가 검색 상위에 뜨는 문제는 metadata/ranking 또는 table handling 쪽 후속 검토 대상이다.

## 적용한 정제 규칙

이번 작업은 사람이 row를 하나씩 편집한 것이 아니라, v0.1.4 계약에 맞춘 규칙을 코드에 반영하고 전체 official corpus에 일괄 적용한 작업이다.

| 규칙 영역 | 적용 규칙 | 이유 |
| --- | --- | --- |
| 원천 데이터 | 기존 `chunks.jsonl`을 기준 source로 사용 | 새 chunk size를 임의로 도입하지 않고 기존 합의 데이터 기준 유지 |
| 표시용 본문 | `text` / `markdown`에는 코드블록, 개행, 표, 원문 출력 예시 보존 | 챗봇 근거 표시와 viewer에서 사람이 읽을 수 있어야 함 |
| BM25/keyword | `normalized_text`는 마크업, 코드 펜스, URL, 과도한 특수문자, 중복 공백 제거 | keyword matching용 flat text 확보 |
| vector input | `embedding_text`는 개행 없는 flat text로 만들고, 표시용 marker와 source/viewer URL 제거 | 임베딩 모델에 문서 서식 노이즈가 들어가지 않게 함 |
| 기술 신호 보존 | `$`, `-n`, `--patch`, `/`, `.`, `:`, `=`, `{}`, `[]` 등 명령어/리소스/YAML 의미 기호는 필요한 범위에서 보존 | OCP 문서에서는 기호가 명령어와 설정의 의미 일부이기 때문 |
| 출력 예시 제거 | `Tk9...` 형태의 padding 있는/없는 base64-like token은 `normalized_text` / `embedding_text`에서 제거 | 질문 의미와 무관한 encoded sample output이 검색 점수를 흐리지 않게 함 |
| 원문 보존 | base64-like output도 표시용 `text`에는 남김 | 원문 코드블록을 viewer에서 확인할 수 있어야 함 |
| 중복/공백 처리 | empty, exact duplicate, contained overlap projection은 Qdrant embedding 후보에서 제외 | 같은 의미의 중복 vector가 top-k를 낭비하지 않게 함 |
| Qdrant payload | `payload.text`는 표시용, `text_fields.embedding_text`는 vector용, `text_fields.normalized_text`는 BM25용 | 표시 계층과 검색 계층 혼선을 막기 위함 |
| BM25 runtime | BM25 index는 `normalized_text`를 우선 토큰화하고, hit 반환은 표시용 `text` 유지 | 검색은 정제 텍스트로, 화면 표시는 사람이 읽는 텍스트로 분리 |
| seed refresh | stale Qdrant point는 payload만 갱신하지 않고 새 `embedding_text`로 vector까지 재업서트 | 기존 collection 재사용 시 stale vector가 남는 문제 방지 |

### 제거 대상과 보존 대상

| 구분 | 예 | 처리 |
| --- | --- | --- |
| 문서 마커 | `[CODE]`, `[/CODE]`, `[TABLE]`, fenced code marker | `normalized_text` / `embedding_text`에서 제거 |
| URL/anchor | docs URL, viewer path, HTML anchor | 검색 계층에서 제거 |
| base64 출력 예 | `Tk9ERUlQX0hJTlQ9MTkyLjAuMCxxxx==`, `Tk9ERUlQX0hJTlQ9MTkyLjAuMC` | 검색 계층에서 제거, 표시용 `text`에는 보존 |
| 명령어 | `$ oc describe pvc <pvc-name> -n <namespace>` | `embedding_text`에서 의미 기호 보존 |
| 리소스명/경로 | `configs.imageregistry/cluster`, `spec.storage.pvc`, `/etc/default/nodeip-configuration` | 검색 계층에서 의미 신호로 보존 또는 토큰화 |
| YAML/JSON 구조 | `apiVersion:`, `spec:`, `{ "migration": null }` | `embedding_text`에서는 의미 구조를 최대한 보존, `normalized_text`에서는 keyword 매칭에 맞게 평탄화 |

### 절차 판단

이 절차의 핵심은 "원문을 깨끗하게 덮어쓰기"가 아니라, 같은 source row에서 목적별 텍스트 계층을 분리하는 것이다.

따라서 검수할 때는 전체 줄 문자열보다 필드별 역할을 기준으로 봐야 한다.

- `text`에 코드블록/개행/base64 출력 예가 보이는 것은 정상일 수 있다.
- `normalized_text`와 `embedding_text`에 같은 노이즈가 남으면 문제다.
- `normalized_text == embedding_text`인 row가 일부 있는 것은 명령어/코드가 없는 일반 설명 또는 표성 본문에서는 정상일 수 있다.

## 추가 변경 판단 근거

### 1. BM25가 `normalized_text`를 우선 토큰화하도록 바꾼 이유

| 항목 | 내용 |
| --- | --- |
| 원래 상태 | BM25 index는 row의 `text`를 토큰화했다. |
| 문제가 된 이유 | 이번 작업으로 `text`는 표시용 markdown/원문 계층으로 복원됐다. 따라서 `text`에는 코드블록, 개행, 표 표현, 원문 출력 예시가 남을 수 있다. 이 값을 그대로 BM25가 토큰화하면 BM25가 검색용으로 정제된 텍스트가 아니라 표시용 텍스트를 보게 된다. |
| 변경한 내용 | BM25 index 생성 시 `text_fields.normalized_text` 또는 `normalized_text`를 우선 사용하고, 값이 없을 때만 기존 `text`로 fallback하도록 수정했다. |
| 기대 이점 | BM25/keyword 검색은 정제된 flat text를 기준으로 동작하고, 검색 결과 화면이나 답변 근거에는 표시용 `text`를 그대로 돌려줄 수 있다. 즉 검색용 텍스트와 표시용 텍스트가 분리된다. |
| 논의 가능 지점 | `normalized_text`에서 어느 수준까지 기호를 제거할지, 명령어 옵션을 BM25용에서도 일부 보존할지 여부는 팀 검색 정책에 따라 조정할 수 있다. |

이 변경은 "BM25 파일을 없앨지"의 문제가 아니라, runtime BM25가 어떤 텍스트 계층을 검색 대상으로 삼을지에 대한 정리다.

#### `bm25_corpus.jsonl` 상태 정리

질문 대상 파일:

```text
corpus/sources/official/imported-gold/gold_corpus_ko/bm25_corpus.jsonl
```

현재 판단:

| 관점 | 판단 |
| --- | --- |
| DB 기반 현재 runtime | 필수 파일이 아니다. `settings.database_url`이 있으면 `BM25Index.from_postgres()`를 사용한다. |
| 이번 수정 후 BM25 검색 대상 | Postgres row에서 만들어진 payload의 `text_fields.normalized_text` / `normalized_text`를 우선 토큰화한다. |
| jsonl fallback | DB URL이 없을 때는 `BM25Index.from_jsonl(settings.retrieval_bm25_corpus_path)` 경로가 아직 살아 있다. 이 경우 `bm25_corpus.jsonl`이 fallback seed 역할을 할 수 있다. |
| 지금 바로 삭제 가능 여부 | 코드/설정/테스트에서 fallback 경로를 제거하기 전에는 "완전히 삭제 가능"이라고 단정하지 않는 것이 안전하다. |
| 정리 방향 | DB 기반 runtime을 표준으로 확정한다면, 별도 회의 후 `bm25_corpus.jsonl`을 legacy/fallback artifact로 명시하거나 생성/참조 제거 작업을 진행할 수 있다. |

따라서 정확한 표현은 다음과 같다.

```text
현재 DB 기반 runtime에서는 bm25_corpus.jsonl이 주 검색 입력이 아니며,
BM25는 DB row의 normalized_text를 우선 토큰화하도록 수정됐다.
다만 DB-less fallback 경로가 남아 있으므로 파일 삭제는 별도 정리 작업으로 다룬다.
```

### 2. compose seed stale refresh가 vector까지 재업서트하도록 바꾼 이유

| 항목 | 내용 |
| --- | --- |
| 원래 상태 | docker compose seed 경로는 `official-gold-import --index --refresh-qdrant-payloads`를 사용한다. 이 중 refresh 경로는 기존 Qdrant point의 payload hash가 달라졌을 때 payload를 덮어쓰는 역할이었다. |
| 문제가 된 이유 | `embedding_text`가 바뀌었는데 payload만 갱신하면, Qdrant에 저장된 vector는 예전 embedding input 기준으로 남을 수 있다. 화면에 보이는 payload는 최신인데 검색 vector는 낡은 상태가 되는 불일치가 생길 수 있다. |
| 변경한 내용 | stale refresh 대상 point는 payload만 overwrite하지 않고, 현재 `embedding_text`로 다시 embedding한 vector와 payload를 함께 upsert하도록 수정했다. |
| 기대 이점 | 기존 collection을 재사용해 docker compose seed를 다시 돌려도, text layer 변경이 vector 검색에도 반영된다. 다음 작업자가 compose 기반으로 환경을 다시 띄울 때 "payload는 최신인데 vector는 이전 것"인 상태를 줄일 수 있다. |
| 논의 가능 지점 | 운영 환경에서 refresh가 많은 point를 재임베딩하면 비용과 시간이 늘 수 있다. 대량 운영에서는 batch size, refresh limit, shadow collection 전략을 별도로 합의할 수 있다. |

이 변경은 새 collection을 항상 만드는 방식이 아니라, 기존 collection을 재사용하는 개발/검증 흐름에서 데이터 불일치를 줄이기 위한 안전장치다.

## 다음 작업자에게 남길 말

이 브랜치는 청킹/임베딩 텍스트 계층과 Qdrant/BM25 경로를 v0.1.4 계약에 맞추는 작업이다.

다음 단계에서 바로 보면 좋은 것은 챗봇 평가셋이다. 예를 들어 `PVC Pending`, `etcd backup`, `MachineConfigPool degraded`, `이미지 레지스트리 ReadWriteMany` 같은 질문으로 top-k 근거와 최종 답변이 맞는지 확인하면 된다.

단, 이 평가는 metadata/rank policy와 LLM generation까지 포함하므로 이 브랜치의 청킹/임베딩 수정만으로 단정하지 않는 것이 좋다.
