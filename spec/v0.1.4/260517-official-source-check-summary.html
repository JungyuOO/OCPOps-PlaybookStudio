260517 Official Retrieval Source Check

이 문서의 목적

이번 확인은 변경된 official corpus / embedding 데이터가 실제 검색 단계에서 기대한 근거 문서를 잘 가져오는지 보기 위한 작업입니다.

J님이 요청한 범위에 맞춰 답변 생성(answer LLM), reranker, UI 답변 문장은 평가하지 않았고, 검색 결과의 source만 확인했습니다. 즉 “챗봇이 최종 답변을 잘했는가”가 아니라 “검색기가 먼저 올바른 근거 문서를 찾았는가”를 본 결과입니다.

결과를 읽을 때는 전체 점수보다 두 가지를 보면 됩니다. 첫째, 기대 문서가 top5 안에 들어왔는지입니다. 둘째, 상위 근거의 embedding_text / normalized_text에 base64, 코드 마커, 표 마커 같은 정제 노이즈가 남아있는지입니다.

판단 원리

이번 테스트는 RAG 전체 답변 품질을 본 것이 아니라, 검색 단계에서 근거 후보가 어디에서 오는지를 본 것입니다. 흐름을 단순화하면 사용자 질문 → 질문 정규화/확장 → BM25 키워드 검색 + embedding vector 검색 → 결과 결합 → top-k 근거 반환 순서입니다.

청크 정제 문제가 1차 원인이라면 보통 embedding_text에 base64, [CODE], [TABLE], markdown fence 같은 노이즈가 남거나, 목차/표 찌꺼기가 의미 본문처럼 검색되거나, 깨진 텍스트가 여러 질문에서 반복적으로 상위 노출되는 증상이 나타날 수 있습니다.

이번 확인에서는 실패 2건의 상위 근거에서 그런 정제 노이즈는 확인하지 못했습니다. 그래서 현재 증거 기준으로는 청크 오염만을 1차 원인으로 단정하기 어렵고, 질문 해석이나 기대 문서 기준도 함께 확인해야 한다고 보았습니다.

다만 이것이 “청크 영향이 전혀 없다”는 뜻은 아닙니다. 후속 확인이 필요하면 실패 질문의 원문 청크, embedding_text, normalized_text, 기대 문서 쪽 chunk 존재 여부, 질문 해석/메타정보 매칭 기준을 나란히 비교하는 방식이 좋겠습니다.

Summary

  • 확인 범위: 변경된 official corpus / embedding 데이터의 retrieval source 결과만 확인했습니다.
  • 제외 범위: answer LLM 답변 생성, reranker 품질, UI 답변 문장은 이번 테스트에서 제외했습니다.
  • 실행 조건: reranker는 끄고(RERANKER_ENABLED=false), 로컬 DB(127.0.0.1:5432)와 Qdrant openshift_docs 기준으로 실행했습니다.
  • 전체 결과: 18개 질문 중 16개는 기대 문서가 top5 안에 포함됐고, 2개는 기대 문서 기준과 실제 검색 결과가 달랐습니다.
  • 지표 해석: hit@1=0.7222, hit@3=0.8889, hit@5=0.8889입니다. 즉 첫 번째 결과부터 맞은 케이스가 약 72%, 상위 5개 안에 기대 문서가 들어온 케이스가 약 89%입니다.
  • reranker 확인: reranker를 끈 조건에서 실행했고, rerank로 top1이 바뀐 케이스는 0건입니다.
  • source 확인: 상위 검색 결과 90개 중 89개는 official docs였고, 1개는 study docs였습니다.
  • 정제 상태 확인: 상위 근거의 embedding_text / normalized_text에서 base64, CODE/TABLE marker, markdown fence, anchor noise는 확인되지 않았습니다.

청크 영향 가능성에 대한 판단

이 2건에 대해 청크 영향 가능성을 완전히 배제하지는 않았습니다. 다만 이번 확인에서는 실패 케이스의 상위 근거에서 base64, 코드 마커, 표 마커, markdown fence 같은 정제 노이즈가 확인되지 않았습니다.

따라서 현재 증거만으로는 “청크 오염 때문에 실패했다”고 단정하기 어렵습니다. 후속 확인이 필요하다면 실패 케이스의 원문 청크, embedding_text, normalized_text, 질문 해석, 메타정보 매칭 기준을 함께 비교하는 방식이 좋겠습니다.

일부 케이스만 기대 문서와 어긋나는 것은 RAG 검색에서 충분히 발생할 수 있습니다. 검색 결과는 청크 품질뿐 아니라 질문의 구체성, embedding 유사도, BM25 키워드 매칭, metadata, source/book 분류 기준이 함께 작동한 결과이기 때문입니다.

Shareable Note

변경 데이터 기준으로 검색 근거 문서만 확인했고 answer LLM/reranker는 제외했습니다. 전체 18건 중 top5 기준 16건은 기대 문서가 포함됐고, 2건은 기대 문서 기준과 실제 검색 결과가 달랐습니다. 상위 근거의 embedding/normalized text에서는 base64나 코드 마커 같은 정제 노이즈를 확인하지 못했습니다. 그래서 이 2건은 제가 임의로 수정하기보다, 질문 의도와 기대 문서 기준을 먼저 맞춰보면 좋겠다고 판단했습니다. 이 방향이 적절한지 의견 부탁드립니다.
