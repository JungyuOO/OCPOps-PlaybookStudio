acceptable partial ldap-auth, operator-concept, 일부 개념/문서 위치 질문 문서군과 근거 방향은 맞고, 자동 landing term이 한국어 표현/동의어 차이로 놓친 케이스. 무리하게 로직을 더 늘리지 않음.

4. 질문을 바꾼 것이 아니라 검색 입력을 바꿨다

중요한 정정: 사용자가 입력한 질문 문장 자체를 고쳐서 통과시킨 것이 아닙니다. 같은 질문을 그대로 두고, retrieval 내부에서 붙는 보조 검색어와 가중치 기준을 보정했습니다. 즉 “질문 바꿔치기”가 아니라 “검색기가 질문 의도를 덜 새게 만든 작업”입니다.
case원 질문수정 전 내부 검색 입력 문제수정 후 내부 검색 입력 방향
etcd-restore OpenShift 4.20에서 etcd 스냅샷으로 클러스터를 복원할 때 restore 스크립트와 절차는 어디에 있어? cluster-backup.sh, oc debug node, chroot /host 같은 백업/노드 확인 신호가 같이 붙어 복원 질문이 흐려졌습니다. cluster-restore.sh, /usr/local/bin/cluster-restore.sh, 이전 클러스터 상태로 복원 신호를 강화하고 /var, lsblk류 노드 파티션 신호를 낮췄습니다.
registry-pvc 이미지 레지스트리 스토리지를 PVC로 설정하려면 configs.imageregistry/cluster에서 어떤 필드를 바꿔야 해? 일반 PVC, PersistentVolumeClaim, StorageClass 쪽 신호가 강해져서 일반 storage 문서가 먼저 올라왔습니다. configs.imageregistry/cluster, spec.storage.pvc, openshift-image-registry, Image Registry Operator 신호를 강화했습니다.
route-create OpenShift에서 서비스를 외부로 노출하기 위해 Route를 생성하는 절차와 oc expose 명령어는 뭐야? Route 일반 설명과 HTTP header route 설정 쪽으로 넓게 검색됐습니다. oc expose, oc expose service, 서비스 노출, 경로 생성 신호를 강화하고 header route 신호를 낮췄습니다.
pod-pending-scheduling Pod가 Pending 상태일 때 스케줄링 실패 원인과 이벤트를 확인하는 절차는 어떻게 돼? Pod Pending 질문이 etcd Pod, S2I build pod, machine-api operator pod 같은 다른 Pod 문제로 새었습니다. FailedScheduling, oc describe pod, oc get events, scheduler/event 신호를 강화하고 etcd/build/operator pod 신호를 낮췄습니다.
oidc-auth 외부 OIDC ID 공급자로 OpenShift 인증을 설정할 때 authentication.config/cluster를 수정하는 절차는 뭐야? OIDC 키워드만 보고 release notes의 확인된 문제 청크가 먼저 올라왔습니다. authentication.config/cluster, oc edit authentication.config/cluster, 외부 OIDC ID 공급자 구성 절차 신호를 강화하고 release notes/bug 성격을 낮췄습니다.
mco-concept Machine Config Operator가 노드 설정과 머신 구성을 관리하는 방식은 어디에 설명돼 있어? MCO 개념 질문인데 oc get nodes, Ready/NotReady, CSV/Subscription 같은 상태 확인 신호가 붙었습니다. Machine Config Operator, MCO, MachineConfig, MachineConfigPool, machine config daemon, machine_configuration 신호를 강화했습니다.
observability-monitoring OpenShift observability와 monitoring 기능은 각각 어떤 목적으로 쓰이는지 설명해줘 관찰/모니터링 질문이 overview의 큰 목록 설명으로 먼저 갔습니다. “각각 어떤 목적”을 개념 비교 성격으로 보고 Observability 정보, Red Hat OpenShift Observability, monitoring/logging/tracing 신호를 강화했습니다.
image-prune OpenShift 이미지 레지스트리에서 오래된 이미지와 태그를 정리하는 pruning 절차는 어디에 있어? 오래된 이미지 정리 질문인데 이미지 태그 추가/삭제와 외부 이미지 문서가 먼저 올라왔습니다. Image Pruner, oc adm prune images, pruning images, 이미지 자동 정리 신호를 강화했습니다.
route-ingress OpenShift Route와 Kubernetes Ingress의 관계와 차이를 설명하는 공식 문서는 어디야? Route/Ingress 차이 질문인데 oc expose, YAML 생성, IngressController 샤딩 절차가 먼저 올라왔습니다. networking_overview, Ingress 및 Route 오브젝트, Kubernetes Ingress, OpenShift Route, 개념 비교 신호를 강화했습니다.
upgrade-precheck 클러스터 업데이트 전에 oc adm upgrade recommend로 사전 점검하는 절차는 어디에 설명돼 있어? 업데이트 사전 점검 질문인데 절차 문서와 release note 명령 소개가 섞여 있었습니다. oc adm upgrade recommend 명령 신호를 유지하면서 updating_clusters 절차 문서와 release notes 명령 소개가 함께 잡히도록 보정했습니다.

  • 데이터 누락으로 보이는 케이스는 이번 pass에서 확인되지 않았습니다.
