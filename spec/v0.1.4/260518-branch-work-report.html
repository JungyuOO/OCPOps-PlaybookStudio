2. 왜 이 작업을 했나

직전 검증에서 일부 평가셋의 expected_book_slugs가 현재 active official corpus와 맞지 않는 문제가 확인되었습니다. 예를 들어 applications 같은 값은 주제로는 익숙하지만, 현재 적재된 official embedding book set 29권에는 없는 book_slug였습니다.

이 상태에서 retrieval 평가를 돌리면 검색기가 맞힐 수 없는 기준으로 채점하거나, 반대로 실제 근거 청크는 약한데 문서명만 맞아서 통과처럼 보일 수 있습니다. 그래서 먼저 평가 기준이 현재 적재 데이터와 같은 기준을 보도록 정리했습니다.

6. 남은 4건은 무엇인가

7. 남은 4건 조치 유형

아래 표는 “다음에 무엇을 고쳐야 하는가”를 기준으로 다시 정리한 것입니다. 네 케이스 모두 정답 후보 청크가 Qdrant에 존재하므로, 현재 단계에서는 데이터 누락이나 청크 오염으로 보지 않았습니다.
case조치 유형근거다음 확인
file-integrity-log keyword/BM25 hybrid ranking 정답 청크 50446a40...가 Qdrant에 있고 integritylog, base64 d, gunzip 신호도 남아 있습니다. 하지만 top5에는 catalog/operator/test 성격 청크가 먼저 올라옵니다. 희귀 키워드가 BM25/hybrid에서 충분히 힘을 받는지 확인합니다. 이 케이스는 expected 보정보다 검색 순위 쪽 조치가 우선입니다.
web-console-overview metadata/title/section 정답 청크 0faa0453..., 5c40fd91...가 Qdrant에 있고 웹 콘솔의 워크로드/메트릭/상태 확인 설명이 있습니다. 그러나 질문이 “어떤 문서에서 설명해?” 형태라 본문 유사도만으로는 support/update 문서가 먼저 잡힙니다. 문서명, book_slug, section_path, title 신호를 검색 점수에 어떻게 반영할지 확인합니다. 이 케이스는 metadata 활용 쪽 조치가 우선입니다.

postinstall-cluster-status expected 기준 ranking 후보 postinstallation 정답 청크 943b0346..., 60382fdc...가 Qdrant에 있고 oc get nodes, oc get ClusterOperators 근거가 있습니다. 다만 top1의 validation_and_troubleshooting도 노드 상태 확인 절차를 일부 포함합니다. validation_and_troubleshooting을 expected에 허용할지 먼저 협의합니다. 그 뒤에도 postinstallation 직접 명령 청크가 밀리면 ranking 문제로 봅니다.

8. 의아할 수 있는 지점 정리

질문답변
이번 브랜치에서 코퍼스나 Qdrant를 다시 고쳤나요?아니요. 이번 범위는 평가셋 계약 정리와 검수입니다. DB/Qdrant 재적재, embedding chunk 수정은 하지 않았습니다.
query rewrite 변경분은 현재 브랜치에 있나요?네. 현재 브랜치에는 dev에서 들어온 query signal/rewrite 계열 변경분이 이미 포함되어 있습니다. 다만 이번 단독 질문형 retrieval 평가는 context가 없어서 명시적 follow-up rewrite는 적용되지 않았고, 결과에는 rewrite_applied=false, rewrite_reason=no_context로 남았습니다.
applications를 왜 없앴나요?주제 자체를 부정한 것이 아니라, 현재 active official embedding book set에는 applications라는 book_slug가 없기 때문입니다. 평가 expected는 현재 검색기가 실제로 맞힐 수 있는 active book 기준이어야 합니다.
그럼 retrieval miss 4건은 평가셋 탓인가요?전부 평가셋 탓으로 보지 않았습니다. 정답 청크 존재 여부를 확인했고, 대부분은 “정답은 있지만 top-k로 못 올라오는 문제” 또는 “허용할 expected 범위가 겹치는 문제”로 분리했습니다.
이번 작업으로 챗봇 답변 품질까지 검증됐나요?아니요. answer LLM, reranker, UI 답변 문장은 이번 범위에서 제외했습니다. J 요청대로 retrieval source와 top chunk 근거 중심으로만 봤습니다.
다음에 무엇을 보면 되나요?현재 포함된 query signal/rewrite 기준으로도 남는 4건을 봐야 합니다. 특히 단독 질문에서 rewrite가 적용되지 않는 구조가 맞는지, 아니면 metadata/section/title 신호나 검색 순위 조정으로 풀 문제인지 나눠서 보면 됩니다.
