260517 청크/임베딩 정리 흐름

이 문서는 상세 로그가 아니라, 오늘 청크/임베딩 텍스트를 왜 다시 봤고, 무엇을 고쳤고, 결과가 어땠는지 빠르게 이해하기 위한 정리본입니다.

먼저 이것만 보면 됩니다

  • 문제: file-integrity 주변 일부 embedding chunk에 base64 tail과 SHA성 꼬리 토큰이 남아 있었습니다.
  • 수정: 원본 chunks.jsonl은 건드리지 않고, 검색용 projection과 생성 규칙을 정리했습니다.
  • 반영: 로컬 DB/Qdrant에는 확인된 2개 청크만 재업서트하고, tail-only 1개 point만 삭제했습니다.
  • 검증: 관련 단위 테스트 13개 통과, projection hard fail 0건, Qdrant 직접 조회 정상.
  • 남은 일: 정답 청크가 있는데도 top-k로 못 올라오는 케이스는 J의 query rewrite 수정분과 함께 다시 비교해야 합니다.

왜 이 문서를 따로 만들었나

기존 260517-official-data-validation.html은 상세 검증 로그입니다. 계속 내용을 붙이다 보니 읽는 사람이 “그래서 어느 시점에 뭘 고쳤다는 거야?”를 바로 파악하기 어려워졌습니다.

그래서 이 문서는 청크/임베딩 텍스트 수정 흐름만 따로 떼어, 이해용 순서로 다시 정리했습니다.

브랜치feat/corpus_embedding_test_S_v.2
범위official corpus / embedding 품질
HEADe77a6b6 + working tree 변경
기준 브랜치 확인: 현재 브랜치는 origin/dev의 c175055를 merge-base로 두고 있습니다. HEAD..origin/dev는 비어 있고, origin/dev..HEAD에는 e77a6b6 한 개가 있습니다.

이 확인의 의미는 원인 단정이 아니라 현재 작업 기준점 확인입니다. 즉 예전 브랜치 잔상을 기준으로 보는 것이 아니라, dev 기준 새 브랜치에서 지금부터 데이터 품질 점검과 보정을 이어가면 된다는 뜻입니다.
중요: corpus/sources/official/imported-gold/gold_corpus_ko/embeddings/embedding_chunks.jsonl은 Git ignore 대상입니다. 그래서 로컬 파일만 고치면 전달성이 없습니다. 이번에는 같은 문제가 다시 생기지 않도록 생성 코드와 테스트도 함께 수정했습니다.

이번 retrieval 테스트 설계

이번 테스트는 챗봇 답변 평가가 아닙니다.
목적은 변경된 official embedding 데이터가 질문에 맞는 근거 문서와 청크를 검색 단계에서 가져오는지 확인하는 것이었습니다. 그래서 answer LLM, reranker, UI 문장은 제외했습니다.
30건단독 질문형 데이터 검증 질문
top5기대 book_slug가 들어오면 문서 기준 통과
별도 확인top chunk 텍스트 직접성, 정제 노이즈

질문지를 이렇게 만든 이유

  • 사용자가 처음부터 물어볼 수 있는 단독 질문으로 만들었습니다. 그거, 아까 같은 멀티턴 표현은 제외했습니다.
  • 각 질문에는 validation_focus를 넣어 “무엇을 검증하려는 질문인지”를 적었습니다.
  • expected_book_slugs는 문서 단위 모범 매칭입니다. 예를 들어 etcd 백업 질문이면 etcd, backup_and_restore, postinstallation_configuration이 모범 범위입니다.
  • expected_landing_terms는 청크 안에 살아 있어야 하는 핵심 단어입니다. 이건 점수 계산의 정답표라기보다 사람이 top chunk를 읽을 때 보는 기준입니다.

4

5

DB/Qdrant 반영

dry-run에서 전체 skipped가 2,026건으로 넓게 잡혔기 때문에, --delete-skipped 전체 삭제는 하지 않았습니다. 대신 확인된 2개 청크만 재임베딩/upsert하고, tail-only 1개 point만 삭제했습니다.

  • 산출물은 “정보를 많이 넣기”보다 “읽는 사람이 결정할 수 있게 만들기”가 우선입니다.
