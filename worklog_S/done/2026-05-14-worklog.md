# 2026-05-14 작업 기록

## 핵심 판단

어제의 중심 작업은 `Wiki Library`와 업로드 파이프라인을 "보여주기 UI"가 아니라 실제 데이터/이벤트/품질 상태를 반영하는 제품 흐름으로 맞추는 일이었다.

완료라고 과장하면 안 되는 부분도 있다. 특히 업로드 품질과 Gold 승격은 아직 런타임 실물 검증을 계속해야 한다.

## 진행한 작업

### Wiki Library IA 재정리

- `Wiki Library`를 상위 화면으로 보고, `Book Factory`를 그 안에서 문서를 생산/보강/검수하는 본체로 재정의했다.
- `System Data Board -> WIKI Scope -> Wiki Library Output -> Book Factory -> Documents` 흐름을 제품 구조로 잡았다.
- `Book Factory`는 drawer가 아니라 메인 섹션이어야 한다는 방향을 확정했다.
- `Bronze -> Silver -> Gold -> Judge`는 장식이 아니라 scope별 실제 생산 상태와 다음 조치를 보여줘야 한다는 기준을 세웠다.

### 업로드 파이프라인 실제화

- 기존 문제: 파일 선택 직후 프론트가 추정값으로 단계를 켜서 실제 사건과 UI가 따로 놀았다.
- 방향: 서버가 기록한 이벤트 원장 기준으로만 단계가 움직여야 한다.
- 관련 커밋 근거:
  - `d147e26 feat: stabilize wiki upload pipeline`
  - `c5d7443 feat: preserve wiki upload pipeline after dev sync`
  - `b327e19 fix: ground upload pipeline in recorded events`
- 핵심 계약:
  - Bronze: 파일 수신/원천 저장
  - Silver: 파싱/chunk/persist
  - Gold: 색인/Gold build
  - Judge: 품질 판정
  - Topology: 지식망 snapshot
- 남은 주의점: 실제 업로드를 끝까지 반복 검증해서 "갑자기 pass처럼 보이는 UI"가 남아 있지 않은지 봐야 한다.

### 데이터 품질 / Gold Gate

- Gold는 "DB에 저장됨"이 아니라 "읽기, 검색, 지식망, 품질 판정 통과"여야 한다는 기준을 세웠다.
- `code_loss`, `page_stub`, 단어 깨짐, 과도한 공백, topology 누락은 Gold blocker 또는 repair 대상이어야 한다.
- 코드블록 자동 수리 방향을 잡았다.
  - YAML/명령어를 deterministic 방식으로 fence 복구
  - 수리 후 chunk 재생성, 재색인, topology 재생성, 품질 재검사
- 남은 주의점: 수리 버튼이 있어도 실제 원문 대비 품질이 나쁘면 Gold로 올리면 안 된다.

### dev 반영 / reranker 정리

- dev 최신 변경을 반영하는 과정에서 리랭커 변경사항을 확인했다.
- 이해한 내용:
  - 업로드/코퍼스 생산 파이프라인 자체는 리랭커가 필수 전제는 아니다.
  - 리랭커는 주로 챗봇 검색/응답 품질 검증 단계에서 중요하다.
  - 서버 쪽 리랭커를 쓰면 로컬 env 필수성은 낮지만, 로컬에서 빠르게 수정/검증할 때는 endpoint/env가 필요할 수 있다.
- 관련 커밋 근거:

  - `4ced4b4 Record Dragonkue reranker smoke evidence`
  - `b48ba3f Align reranker defaults with Korean model`
